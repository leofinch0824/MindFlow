# MindFlow 交互审查设计文档（用于研发可执行版 PRD 前置）

> 日期：2026-04-12
> 状态：已确认设计（待执行）
> 产物类型：Brainstorming 设计规格（非实现）
> 目标读者：产品经理、研发负责人、前端/后端工程师

---

## 1. 背景与问题

在进入“研发可执行版 PRD”编写之前，需要先完成一轮结构化交互审查，避免在已有实现与文档口径不一致的情况下直接写执行文档，导致研发按错误前提推进。

当前主要风险：
- 文档定义、前端行为、后端接口之间可能存在偏差
- 某些交互虽然可用，但不符合需求定义或体验预期
- 缺少“已合理 / 不合理”的分层清单，难以组织后续需求澄清

---

## 2. 审查目标

本轮审查要达成以下目标：

1. 形成“页面交互 -> 前端触发 -> 后端接口 -> 需求条目”的完整证据链映射
2. 判定每个主要交互逻辑是否合理（产品一致性 + 工程可执行性 + 体验合理性）
3. 输出两类清单：
   - 合理交互逻辑（可复用）
   - 不合理交互逻辑（需深挖）
4. 产出“深挖议题列表”，作为后续需求讨论输入
5. 为研发可执行 PRD 提供准确、可落地的前置事实基础

---

## 3. 判定基准（已确认）

采用联合判定基准：

- 主文档：`docs/superpowers/specs/2026-04-10-product-requirements.md`
- 辅文档：`docs/superpowers/specs/2026-04-09-ai-news-digest-v2-technical-spec.md`

冲突处理规则：
- 若 PRD 与技术规格冲突，**以 PRD 为准**
- 同时将冲突本身记录为“规格矛盾风险”

---

## 4. 审查范围（已确认）

采用全量 C 范围：

1. 已实现且可在前端触发的交互（按钮、输入、切换、表单提交等）
2. 文档声明但前端未落地的交互（缺失项）
3. 后端已存在但前端未接入的接口能力（接入缺口）

明确排除：
- 本轮不做代码修改、不做重构、不做功能新增实现
- 本轮不定义最终 UI 视觉方案

---

## 5. 判定视角与等级（已确认）

采用体验优先（C）：
- 产品一致性：是否符合需求定义的用户行为与业务目标
- 工程可执行性：接口契约、状态处理、错误分支、数据口径是否闭环
- 体验合理性：交互反馈是否及时明确、是否符合常见可用性实践

### 5.1 判定结果等级

- **Green（合理）**：需求一致 + 工程闭环 + 体验无明显断裂
- **Yellow（待完善）**：主流程可用，但边界条件或反馈机制不足
- **Red（不合理）**：与需求冲突、契约不闭环或造成关键误导

### 5.2 优先级

- **P0**：影响核心闭环或会导致错误实现
- **P1**：影响主要任务效率与可理解性
- **P2**：优化项，不阻塞主流程

---

## 6. 推进方案对比与决策（已确认）

### 方案 1：静态对照审查
- 仅对代码和文档做映射与判定
- 优点：快
- 风险：对运行时行为可信度不足

### 方案 2：证据链审查（推荐并已确认）
- 先静态映射，再对关键链路做重点验证
- 优点：效率与可信度平衡最佳
- 适用：当前项目阶段（需要先建立可靠事实基础）

### 方案 3：全量端到端实测审查
- 对全部交互做运行态验证
- 优点：证据最强
- 风险：成本高、受环境依赖强

最终采用：**方案 2 + 高风险链路点测增强**

---

## 7. 执行流程与里程碑（已确认）

1. **基线采集**
   - 汇总页面交互点、前端 API 调用、后端路由、需求条目引用
2. **证据链校验**
   - 对关键链路补充验证（简报、兴趣、信息源、AI 配置、行为反馈）
3. **三维判定**
   - 依据产品/工程/体验三维打 Green/Yellow/Red + P0/P1/P2
4. **问题归并**
   - 合并重复问题，保留最小问题集
5. **结论交付**
   - 形成合理/不合理双清单 + 深挖议题列表

---

## 8. 交付结构（已确认）

最终交付为一个审查结论包（单文档分章节）：

### 8.1 交互全景映射表（主表）
字段：
- 页面
- 控件/触发入口
- 用户动作
- 前端触发逻辑
- 后端接口
- 需求条目依据
- 当前行为状态
- 判定结果（Green/Yellow/Red）
- 优先级（P0/P1/P2）

### 8.2 合理交互逻辑清单
- 只列可作为后续规格模板的交互
- 说明“为什么合理，复用边界是什么”

### 8.3 不合理交互逻辑清单（重点）
每条包含：
- 现状行为
- 偏差点（相对需求）
- 工程/体验风险
- 可执行修正建议（交互规则 + 接口建议）
- 优先级

### 8.4 深挖议题列表
- 从 Red/P0-P1 问题中抽取“需产品决策的问题”
- 作为下一轮需求讨论输入

---

## 9. 质量门槛（审查结果验收标准）

本轮审查完成的判定标准：

1. 所有主要可交互功能都有证据链记录
2. 每个 Red/Yellow 问题都能定位到明确证据与需求条目
3. 不合理清单中的建议具备可执行性（不是泛化建议）
4. 深挖议题可直接用于下一轮需求澄清会议
5. 结论可直接支撑“研发可执行版 PRD”编写

---
