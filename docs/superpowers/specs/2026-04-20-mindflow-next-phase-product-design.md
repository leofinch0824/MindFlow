# MindFlow 下一阶段产品设计摘要

> 日期：2026-04-20
> 状态：draft-for-review
> 目标：明确 MindFlow 从当前 `digest-first` MVP 进入下一阶段后的产品目标、页面角色、核心链路与 MVP 范围，防止后续开发继续围绕“日报体验优化”单线扩张而偏离产品主方向。

---

## 1. 产品定义

### 1.1 一句话定位

MindFlow 下一阶段定义为：

**一个以 `Daily Digest + Now` 为双核心入口的私有化信息工作台。**

### 1.2 产品本质

MindFlow 不是传统 RSS 阅读器，也不是把所有内容按时间倒序堆叠的信息流产品。

它的核心目标是：

- 让用户围绕自己真正关心的信息源获取高密度、低噪音的信息输入
- 通过 `Daily Digest` 提供每天的重点编排结果
- 通过 `Now` 承接具体内容的深入阅读与处理
- 基于信息源、兴趣标签与阅读行为逐步调整内容优先级

### 1.3 设计原则

- `Daily Digest` 是核心入口，但不是唯一内容容器
- `Now` 是工作台，不是普通时间流
- 产品优先帮助用户判断“现在该看什么”，而不是展示“今天抓到了什么”
- 尽量把用户停留在系统内部完成理解与处理，再决定是否跳到原文

---

## 2. 页面结构与角色

### 2.1 页面角色

下一阶段先收敛为以下核心页面或视图：

#### Daily Digest

- 默认首页
- 保留更强的刊物式阅读感
- 继续沿用 MindFlow 当前的 UI 配色和内容气质
- 保留显式兴趣纠偏能力
- 作为进入具体内容详情的主入口

#### Now

- 当日优先级内容工作台
- 展示系统判断“当前值得处理”的内容列表
- 承接从 `Daily Digest` 进入的具体文章
- 不承担长期历史归档职责

#### Detail

- 作为 `Now` 体系中的主消费场景
- 用于完整展示具体文章内容
- 承担系统内阅读与处理动作
- 不承担显式兴趣纠偏

#### Sources / Interests / Settings

- 继续保留
- 不再只是日报生成的辅助配置页
- 是整个信息工作台的控制面板

### 2.2 页面关系

- `Daily Digest` 与 `Now` 共同构成双核心入口
- `Detail` 是 `Now` 的内容消费主场
- `Sources / Interests / Settings` 提供底层控制能力

---

## 3. 核心交互链路

### 3.1 主链路

下一阶段的主阅读链路正式定义为：

**`Daily Digest -> Detail -> (可选) Read Source`**

### 3.2 交互规则

- 用户从 `Daily Digest` 的内容卡进入详情页
- 不再从日报卡片直接把用户大量导向外部原文
- 详情页中先展示系统内的完整阅读内容
- `Read Source` 保留，但降级为详情页中的次动作
- 用户是否打开原文，由详情页阅读后自行决定

### 3.3 产品意图

这条链路的目标是：

- 延长用户在 MindFlow 内的停留时长
- 让系统内部的加工价值先被消费
- 让原文跳转成为主动选择，而不是默认逃逸路径

---

## 4. Now 与 Detail 的产品定义

### 4.1 Now 的定位

`Now` 在 MVP 中不是普通文章列表，也不是完整内容归档库，而是：

**一个面向“当天/近期值得处理内容”的优先级工作台。**

### 4.2 Now 的职责

`Now` 的职责只包含三类：

- 承接从 `Daily Digest` 进入的具体文章
- 展示系统判断“当前值得看”的内容列表
- 提供最小阅读处理动作

### 4.3 Now 的内容边界

`Now` 的内容范围建议先定成：

- 以当日内容为主
- 可带少量最近 24-48 小时内仍高优先级但未处理的内容
- 不承担长期历史内容展示
- 不承担完整归档检索

### 4.4 Detail 的内容结构

详情页中，`AI 摘要 / 洞察` 与 `清洗后的正文` 必须在**同一个详情页面**里呈现。

结构建议如下：

1. 标题 / 来源 / 标签 / 时间
2. 简短 AI 摘要高亮块
3. 清洗后的正文内容
4. 已读 / 已处理动作
5. 可选 `Read Source`

其中：

- AI 摘要是正文前的一小段高亮块
- 不是单独页面，也不是单独 tab
- 正文仍是详情页的主体

---

## 5. Detail 与 Now 的布局关系

### 5.1 总体体验

从用户感知上：

- `Daily Digest` 保持更强的刊物阅读体验
- `Now` 使用工作台结构
- 两者属于同一个产品系统，不是割裂产品

### 5.2 工作台布局

`Now` 采用三栏工作台方案的产品心智，但在 MVP 中优先保证“详情阅读优先”：

- 左侧：视图或上下文入口
- 中间：当前优先级内容列表
- 右侧：文章详情

### 5.3 MVP 交互约束

MVP 中：

- 默认以详情阅读为主
- `Now` 列表可以展开/收起
- 列表不是永远强占屏幕
- 用户始终知道当前文章处于“今日优先级上下文”里

这样可以兼顾：

- 刊物感后的沉浸阅读
- 工作台里的上下文感

---

## 6. 显式反馈与学习规则

### 6.1 显式兴趣纠偏的位置

显式兴趣纠偏只保留在 `Daily Digest`：

- `Daily Digest` 负责“编排层反馈”
- `Detail` 不放显式兴趣纠偏

### 6.2 原因

如果 `Detail` 继续复用 `Daily Digest` 的兴趣纠偏按钮，会导致：

- 语义冲突
- 同一内容重复加权
- 阅读层与编排层角色混乱

因此在 MVP 中明确禁止：

- 在详情页继续复用 Digest 的兴趣纠偏动作

### 6.3 后续扩展原则

未来如果 Detail 需要承载偏好控制，必须使用更高层级语义，例如：

- 减少这个来源
- 这个主题更重要
- 类似文章优先级降低

但这不进入当前阶段 MVP。

## 7. MVP 功能清单

### 7.1 必须进入下一阶段 MVP 的能力

#### Daily Digest

- 保持首页定位
- 保持刊物式呈现
- 保留显式兴趣纠偏
- 每条内容可进入详情页

#### Now

- 优先级内容列表
- 承接从 `Daily Digest` 进入的具体内容
- 可展开/收起列表

#### Detail

- 展示 AI 摘要高亮块
- 展示清洗后的正文
- 打开原文
- 标记已读 / 已处理

#### 系统底层

- 继续复用当前 Source / Article / Anchor / Digest 数据链路
- 复用当前兴趣标签和行为信号体系
- 继续以内容优先级排序驱动 `Now`

### 7.2 明确不进入当前阶段 MVP 的能力

- 稍后读
- 收藏
- 归档
- Backlog 独立视图
- Archive 独立视图
- Detail 页显式兴趣纠偏
- 复杂 explainability
- 来源权重手动调节 UI
- 多级批处理动作

---

## 8. Now 的优先级逻辑

### 8.1 产品层要求

MVP 阶段，`Now` 不需要做成复杂黑箱推荐系统，但必须具备稳定排序逻辑。

优先级由以下信号共同决定：

- 来源相关性
- 兴趣标签匹配度
- AI 提炼出的显著性
- 时效性
- 处理状态

### 8.2 产品层约束

当前阶段不要求：

- 向用户完整解释排序公式
- 提供复杂的人工调权 UI

当前阶段只要求：

- 用户能感受到 `Now` 不是乱排的
- 用户使用后，后续排序会逐步变化

---

## 9. 对现有能力的重新定位

### 9.1 DailyDigest

从“唯一核心产品容器”调整为“双核心之一”。

### 9.2 InsightCard

从阅读终点改为详情页入口。

### 9.3 Article

从后台数据记录提升为前台核心消费对象。

### 9.4 Sources

从抓取配置页升级为内容优先级体系的底层输入面板。

### 9.5 AI

继续保留摘要和洞察能力，但产品体验重点改为“帮助阅读”，而不只是“服务日报生成”。

---

## 10. 结论

MindFlow 下一阶段不是做“更复杂的日报”，而是：

**在保留日报刊物感的前提下，把日报延伸成一个能承接具体内容阅读的个人信息工作台。**

这一阶段的关键不是扩更多复杂功能，而是把产品主线从：

`Digest-first 单容器`

演进为：

`Daily Digest + Now 双核心`

并通过 `Detail` 把内容消费完整留在系统内部。
